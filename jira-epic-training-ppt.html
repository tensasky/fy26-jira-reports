01 / STRUCTURE

Jira层级结构

🏢 Initiative (项目集)
↓
📁 Feature (子项目)
↓
📦 Epic (任务包)
↓
📝 Story / Task
💡 关键概念
当Intake进入Delivery阶段时，必须通过Jira进行管理。Goal-Initiative-Feature是固定的父子关系，Epic及以下在各自团队Space中管理。
02 / EPIC TYPES

Epic类型与命名

1

Dev Delivery

命名规范：

Dev & SIT - Application Name

  • 每个Epic为实际部署交付的最小单位
  • Application级别，对应PBN中的B
  • Epic工单号需与CAB变更工单号关联
  • 多个部署团队会有多个Epic，以后缀区分
2

Project Management

文档类前缀：

BRD PRD ISD FSD CSBD SRE Solution

里程碑类前缀：

TUAT BUAT Penetration Test Cutover Plan Hypercare

03 / EPIC FIELDS

Epic必填字段 (13项)

No 字段名称 负责人 填写说明
1 Space 系统 Jira空间名字。常用名：板子、Space。每个Space都有字母缩写如CNTIN、CNTD、CNDIN等
2 Parent No PM 当前Ticket的Parent（父项）。Feature-Epic是固定的父子关系，请勿随意修改
3 Jira No 系统 由Space缩写+数字组成，如CNTIN-1234
4 Summary PM 参考Epic Breakdown中的建议前缀+个性化自定义命名
5 Assignee PM 实际管理人和操作人，对于Feature来说是项目的PM。请勿随意修改
03 / EPIC FIELDS (续)

Epic关键字段详解

📄 Description 描述

格式固定：

  • 1. Risk log 表格
  • 2. 项目文档存储地址

文件的确认版需要存档至Epic父项Feature的Description内

📅 日期字段

Start date - 计划开始时间

Due date - 计划结束时间

🔗 Linked work items

涵盖所有非父子关系的关联Ticket：

  • Related to: 用来表示Intake，由TPMO管理
  • Dependency: 用来表示依赖项，由PM管理

📊 Status & Status Trend

Status（状态）：

New → In Progress → Done

Status Trend（趋势）：

根据项目实际状态选择，由Program Manager维护

04 / STORY FIELDS

Story/Task字段要求

No 字段名称 必填 填写说明
1 Space 系统 Jira空间名字
2 Parent No 是 Epic-Story/Task是固定的父子关系，在Parent字段进行关联，请勿随意修改
3 Summary 是 根据任务实际需要自定义命名
4 Assignee 是 实际管理人和操作人，对于Feature来说是项目的PM
5 Description 是 格式固定：1. Risk log表格 2. 项目文档存储地址
6 Start/Due date 是 计划开始和结束时间
7 Status 是 New(未启动) → In Progress(进行中) → Done(业务上线)

07 / BEST PRACTICES

最佳实践与FAQ

✅ Do's - 应该做

  • 及时更新Status状态
  • 准确填写开始/结束日期
  • Description中包含Risk log
  • 关联正确的Parent工单
  • CAB变更工单号关联到Epic
  • 文档确认版存档至Feature

❌ Don'ts - 不要做

  • 不要随意修改Parent字段
  • 不要遗漏必填字段
  • 不要在Epic下直接创建Task（应先建Story）
  • 不要将不相关的Ticket链接为Dependency
  • 不要延迟更新Done状态

🛠️ 字段设置
